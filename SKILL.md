# AIOps 故障诊断

将输入资料转化为基于证据的运维诊断报告。不要把关键词匹配当成确定根因，也不要编造输入中没有的设备、时间、服务、IP、指标或日志。

## 核心流程

严格按以下顺序执行：

1. 识别资料类型和可用信息。
2. 解析或汇总结构化数据。
3. 提取异常事件、影响对象和时间线。
4. 判断故障类型与风险等级。
5. 构建引用原文的证据链。
6. 匹配最相关的故障知识。
7. 生成并排序 2-3 个根因假设。
8. 设计按优先级排列的验证步骤。
9. 给出条件化修复建议与预防措施。
10. 明确证据限制和需要补充的信息。
11. 按固定结构输出 Markdown 报告。

## 识别输入

将输入归入以下一种或多种类型：

- 原始日志：LOG、TXT、系统日志、应用日志、设备日志、数据库日志。
- 告警表：CSV、XLSX、监控平台导出表、巡检表。
- 运维报告：文本型 PDF、巡检报告、健康报告、安全或网络测试报告。
- 故障复盘：DOCX、事故分析、处理记录。
- 用户描述：自然语言故障现象。
- 系统中间结果：异常列表、统计结果、风险等级、RAG 案例。

先说明资料类型。若只有文件路径且可以读取文件，先使用当前环境可用工具提取内容；扫描版 PDF 无文本时说明需要 OCR，不得假装已经读取。

## 按类型处理

### 原始日志

- 解析时间、级别、设备或服务、消息。
- 聚合 ERROR、WARN、Critical 及高频异常。
- 识别同一设备短时间重复事件和跨组件关联。
- 保留原始日志行作为证据。

### CSV 或 Excel

- 优先识别时间、级别、设备、消息字段。
- 接受常见同义字段：`time/date/时间`、`severity/status/级别`、`host/server/node/设备`、`content/detail/description/告警内容`。
- 统计高频告警、高风险设备、时间集中度和故障类型。
- 无法映射时按非结构化文档处理，并明确字段限制。

### PDF 或 Word

- 提取风险关键词、异常段落、设备、服务、指标、处置记录和时间线。
- 若包含足够标准日志行，同时执行日志流程。
- 巡检报告侧重风险与整改；故障复盘侧重时间线、根因证据和改进项。
- 只能引用实际提取到的文本。

### 用户故障描述

- 提取现象、对象、时间、影响和最近变更。
- 资料不足时仍可给出初步假设，但必须标为待验证。
- 只追问能显著区分根因的关键信息。

## 提取与分类

优先关注：

`error`、`failed`、`timeout`、`unavailable`、`packet loss`、`critical`、`exception`、`unreachable`、`permission denied`、`slow query`、`CPU`、`memory`、`disk`、`database`、`port`、`response time`

以及：

`错误`、`失败`、`超时`、`不可用`、`丢包`、`严重`、`异常`、`不可达`、`权限`、`慢查询`、`CPU过高`、`内存过高`、`磁盘异常`、`数据库连接失败`、`端口异常`、`接口响应慢`

优先从以下类别中选择，可在证据不足时使用“未知故障”：

- 连接超时
- 网络丢包
- 网络链路异常
- CPU 资源异常
- 内存资源异常
- 磁盘异常
- 数据库异常
- 数据库慢查询
- 服务不可用
- 接口响应慢
- 端口异常
- 执行失败
- 配置错误
- 权限异常
- 依赖服务异常
- 未知故障

需要具体故障知识、验证命令和修复方向时，读取 [references/fault-knowledge.md](references/fault-knowledge.md)。只加载与当前故障相关的条目。

## 评估风险

- 高：服务不可用、数据库连接失败、严重丢包、磁盘读写失败、多个关键组件同时异常、明确业务中断，或多条高危 ERROR 集中出现。
- 中：WARN 持续出现、CPU/内存持续偏高、接口响应慢、慢查询、单设备异常但未确认中断。
- 低：少量孤立异常、没有明确业务影响、暂不影响核心服务。
- 未知：缺少时间、对象、错误内容或影响信息，无法合理分级。

风险等级必须附带输入证据，不得只按单个关键词机械判定。

## 构建证据链

每条关键证据尽量包含：

- 时间或文档位置
- 设备、服务或业务对象
- 日志级别或指标
- 原始异常内容
- 它支持或反驳的故障假设

引用最少且足够的证据，避免堆砌重复日志。区分：

- 事实：输入直接提供。
- 推断：由多条证据共同支持。
- 假设：需要进一步验证。

证据不足时明确写出：“当前证据不足以确定唯一根因。”

## 生成根因与行动

给出 2-3 个根因假设并按可能性排序。每个假设必须包含：

- 可能性：高、中高、中、低。
- 判断依据：引用具体证据。
- 冲突或缺失证据。
- 最小验证动作：用什么命令、指标或日志验证。

排查步骤按“低风险、高信息增益、快速缩小范围”排序。避免只写“检查网络”或“检查系统”，应写明对象、命令、指标和预期结果。

修复建议必须使用条件句，例如“如果确认端口未监听，则……”。不要在根因未确认时建议直接重启、删除数据、修改生产配置或执行其他高风险操作。

## 与 NetOps Copilot 配合

当输入包含 NetOps Copilot 产生的结构化结果时：

1. 复用其解析、统计、异常分数、故障分类和 RAG 案例。
2. 不把异常分数直接等同于故障严重度。
3. 检查分类结果是否与原始消息一致。
4. 使用原始日志和文档片段建立证据链。
5. 将相关案例视为参考，不视为当前故障事实。

## 输出

默认输出完整 Markdown 诊断报告。读取 [references/report-template.md](references/report-template.md) 并遵循其结构。

若用户只问一个局部问题，可缩短输出，但仍需保留：

- 结论
- 关键证据
- 不确定性
- 下一步验证动作

输出前使用 [references/quality-checklist.md](references/quality-checklist.md) 自检。

## 本地 PoC

需要演示可运行版本时，执行：

```powershell
py -3 scripts/server.py
```

然后访问 `http://127.0.0.1:8765`。该 PoC 支持日志粘贴与文本文件读取、敏感信息脱敏、异常聚合、根因假设排序、Markdown 报告导出和本地审计记录。

运行测试：

```powershell
py -3 -m unittest discover -s tests -v
```

PoC 仅用于诊断辅助：不得自动执行报告中的验证或修复命令，不得将其描述为已接入企业生产环境的成熟平台。
